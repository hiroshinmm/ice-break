# 実装計画: アーカイブ機能（過去ログ閲覧）の構築

## 目的
毎日生成される最新のテックニュースとインサイトをリポジトリ内に永続化。Web ギャラリーから過去分をいつでも振り返られるようにし、情報の資産価値を最大化する。
また、GitHub リポジトリの容量を占有しないよう、古い画像データの自動クリーンアップ機能を組み込む。

## フェーズ 1: データ永続化の基盤 (Data Infrastructure)
### 1.1 データの保存構造 (`src/generateInsights.js`)
- 今日の日付を `Asia/Tokyo` で取得し、`data/archives/YYYY/MM/DD.json` という階層フォルダを作成。
- 保存データに `displayDate` フィールド（例: `2026/03/20`）を付与。
- 現在の `data/insights.json`（最新版用）も並行して従来通り書き出し。
### 1.2 GitHub Actions ワークフロー (`daily-tech-briefing.yml`)
- 生成された JSON ファイルをリポジトリにコミットし、自分自身に `push` するステップを追加。
- 実行時に `git config` でボットのアカウント情報をセットアップ。

## フェーズ 2: Web ギャラリーの拡張 (Frontend)
### 2.1 日付ベースの読み込みの実装 (`index.ejs`)
- クライアントサイド JS で URL パラメータ（`?date=YYYY-MM-DD`）を解析。
- 最新版が表示される基本構造は維持しつつ、パラメータがある場合は `data/archives/...` から JSON を非同期で取得（`fetch`）し、中身を差し替える。
### 2.2 UI の構築
- ヘッダーに「日付を選択」リストを設置（直近 10日分など）。
- 「過去のブリーフィング一覧へ」といったリンクボタンを設置。

## フェーズ 3: リソース管理と最適化 (Maintenance)
### 3.1 画像データの管理ロジック (`src/generateGallery.js`)
- `dist/images/` フォルダにあるファイルの最終更新日時をチェック。
- 14日以上経過した画像ファイルを削除。
- JSON 内の画像パスが存在しない場合に備え、フロントエンドで `originalImageUrl` をフォールバックとして使用（オリジナルのサイトから直接画像を表示）。

## 運用上のメリット
- **情報の継続性の確保**: 「昨日の記事」も URL 一つで振り返り、共有可能に。
- **SEO/シェアの向上**: 各日の記事が固有の URL を持ち、将来的な SNS 連携や SEO に強くなる。
- **永続的な運用能力**: 自動クリーンアップにより、GitHub の無料枠制限を回避。
